배경 분석

AI는 더 이상 챗봇이 아니다

엔터프라이즈에서 AI 도입은 실험 단계를 완전히 벗어나, 대규모 프로덕션 배포 단계로 진입했다. 핵심 변화는 AI의 본질 자체가 달라졌다는 점이다.

1단
단순 챗봇 (Simple Chatbot)
Where / What / Why 같은 단순 질문에 단순 답변. 단발성 요청 처리.
2단
추론형 AI (Reasoning AI)
응답 전에 복잡한 추론 수행. 사고 과정이 내재화된 형태.
3단
🔴 자율 디지털 노동자 (Agentic Workforce)
멀티스텝 워크플로 실행, 코드 작성, 도구 활용, 자율적 작업 수행. AI가 실제로 일한다.
"AI는 이제 단순한 챗봇이 아니라, 멀티스텝 워크플로를 실행하고, 코드를 작성하며, 자율적 디지털 노동력으로 기능하고 있습니다." — 세션 발표자, NVIDIA

이 전환은 인프라 부담을 근본적으로 바꾼다. 단순 질의응답과 달리, 에이전틱 추론은 대규모 지속적 추론 워크로드(sustained inference workload)를 만들어 낸다. 클러스터 전체가 쉬지 않고 가동되어야 한다는 의미다.

아키텍처 개요

젠슨 황의 "5층 케이크"

젠슨 황은 AI 인프라 스택을 5개 계층으로 시각화했다. 원자재 에너지와 칩에서 시작해, 산업 규모의 AI 인텔리전스가 응용프로그램을 구동하는 구조다.

🔑 핵심 포인트

OCI와 같은 CSP(클라우드 서비스 제공자)는 이 5층 케이크의 중간 레이어—네트워킹과 오케스트레이션—를 담당한다. GPU를 에너지로부터 토큰(AI 출력)으로 효율적으로 변환하는 역할이다.

운영 원칙

AI 팩토리 운영의 3대 필수 요소

요소 01
⚡ 성능 가속 (Performance)
GPU가 인프라 오케스트레이션 부담을 지면 안 된다. 레이턴시·지터·대역폭을 타이트하게 제어해 최대 처리량을 보장해야 한다.
GPU 정지(stall)는 토큰 생산과 경제성에 직접 타격을 준다. 따라서 네트워킹·스토리지·보안 워크로드는 GPU 밖으로 완전히 분리(offload)되어야 한다.
요소 02
🛡️ 효율·복원력 (Efficiency & Resiliency)
인프라 워크로드(네트워킹·스토리지·보안)와 AI GPU 워크로드를 명확히 분리. GPU는 오직 토큰 생산에만 집중해야 한다.
오케스트레이션 오버헤드는 GPU 사이클을 낭비한다. Bluefield DPU가 인프라 계층 전체를 흡수함으로써 GPU는 완전히 해방된다.
요소 03
🔐 멀티테넌시·보안 (Multi-tenancy & Security)
비용 효율을 위해 인프라는 여러 테넌트가 공유해야 한다. Zero Trust는 필요조건이지만, 충분조건은 아니다.
단순한 기업 간 분리가 아니라 보안 도메인 단위의 격리가 필요하다. 네트워크 아키텍처에 하드와이어된 격리 프리미티브가 핵심이다.

NVIDIA 솔루션

Bluefield DPU — AI 팩토리의 CPU

NVIDIA Bluefield DPU(Data Processing Unit)는 AI 팩토리의 운영체제를 구동하는 프로세서다. 네트워킹, 스토리지, 보안 워크로드를 호스트 CPU·GPU 대신 DPU가 처리한다.

🌟 ASTRA 솔루션 (Bluefield 4 신기능)

Advanced Secure Trusted Resource Architecture

  • 백엔드 네트워크의 전체 컨트롤 플레인을 DPU로 오프로드
  • 혼잡 제어(congestion control), 멀티테넌시 메커니즘, 네트워크 드라이버까지 DPU에서 실행
  • 호스트 CPU의 테넌트 트래픽 간섭 제거 → 진정한 베어메탈(bare metal) 제공 실현
  • Rubin 세대 시스템에 하드와이어드 내장, OCI에 배포 중

OCI 솔루션

Oracle Acceleron의 3대 기둥

OCI는 2016-17년 출범 시 Gen2 베어메탈 아키텍처를 도입했다. 그러나 AI 혁명은 네트워킹 중심의 근본적 재설계를 요구했고, 그 결과가 Oracle Acceleron이다.

① ConvergeNIC

OCI의 Gen2는 서버 외부에 가상화 장치를 두는 혁신적 설계였다. 호스트 NIC ↔ 오프박스 가상화 장치 간에는 단순 이더넷 프레임만 오고 가므로, 보안 위협 면적이 극히 작다.

ConvergeNIC은 이 두 장치를 하나의 칩으로 통합하되, 각 구역을 하드웨어 수준으로 격리된 FIFO 큐로 연결한다. 보안 이점은 유지하면서 전력·성능 페널티를 제거한다. NVIDIA와의 공동 하드웨어 작업으로 구현.

② 멀티플레너 네트워크 (Multi-Planar Network)

NIC 자체를 1단계 소형 스위치로 활용하여 네트워크 구조를 재설계. 기존 2~3티어 클로스(Clos) 패브릭 대신 k개의 독립 평면(plane)을 병렬 운용한다.

📡 ZetaScale 10

16 제타플롭(ZetaFlops)의 클러스터 컴퓨팅. 기가와트급 캠퍼스 전체에 걸쳐 수십만~수백만 GPU를 단일 거대 클러스터처럼 운영. 훈련 작업 하나가 여러 건물을 가로질러 실행 가능.

③ Zipper — Zero Trust 패킷 라우팅

기존 방화벽 규칙은 보안 정책과 네트워크 정책을 뒤섞는다. 보안 관리자와 네트워크 관리자 양쪽이 동시에 서명해야 하는 구조적 마찰이 있다.

기존 방화벽 (ACL/Firewall Rules)
  • IP CIDR, 포트, 프로토콜 기반 규칙
  • 보안 정책 = 네트워크 정책 혼재
  • 보안 관리자가 마음대로 규칙 변경 시 네트워크 장애 위험
  • 네트워크 관리자가 라우팅 변경 시 보안 구멍 가능
  • 두 부서의 공동 승인 필요 → 운영 병목
Zipper — Zero Trust Packet Routing
  • 리소스에 태그 기반 보안 정책 적용
  • 네트워크 아키텍처와 완전히 독립적인 보안 레이어
  • 예: "이 리소스는 인터넷과 통신 불가" — 라우팅 규칙 무관하게 강제
  • 보안 관리자가 네트워크 지식 없이도 정책 설정 가능
  • 두 관리자 역할의 명확한 분리(decoupling)
Q&A 핵심 인사이트

추론 워크로드와 클러스터 사이즈

Q&A에서 핵심 질문이 나왔다: "추론이 주류가 되면 클러스터를 어떻게 설계해야 하나?"

OCI의 답변 요약
  • 캠퍼스 규모의 클러스터는 훈련이 주 목적, 빌딩 단위(~10만 GPU)는 추론에도 매우 효율적
  • 에이전틱 추론은 단일 GPU 추론이 아님 — 복잡한 멀티스텝 추론 루프 필요
  • SaaS 워크플로 하나를 완성하기 위해 수십~수백 회의 추론 요청이 연쇄됨
  • 추론은 메모리·컴퓨트·로컬 스토리지 요건이 훈련과 다름 → 하드웨어 스펙 조정 필요
  • Bluefield N/S 인터페이스: 에이전틱 추론은 북남(North-South) 대역폭을 폭발적으로 늘림 → 800Gbps NIC 도입 배경
