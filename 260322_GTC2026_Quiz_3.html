01
객관식 · 단일 선택
● 쉬움

NVIDIA가 KV 캐시를 위해 새롭게 정의한 스토리지 계층의 번호는 무엇인가?

정답: C — G3.5 (CMX)
NVIDIA는 기존 G1~G4 계층 사이, G3와 G4 사이에 G3.5라는 새로운 계층을 정의했다. 이것이 바로 CMX(Context Memory eXtension)이며, POD 레벨에서 여러 GPU가 공유 접근하는 KV 캐시 전용 스토리지다.
02
객관식 · 단일 선택
● 쉬움

100,000개(100K) 토큰의 컨텍스트를 가진 KV 캐시 하나의 크기는 대략 얼마인가?

정답: B — 50 GB
발표에서 명시된 수치로, 100K 토큰 컨텍스트 = 약 50GB, 400K 토큰 컨텍스트 = 약 200GB의 KV 캐시 메모리가 필요하다. 이는 단일 에이전트 기준이며, 수백 개의 에이전트가 동시에 동작하면 테라바이트 단위가 된다.
03
객관식 · 단일 선택
● 보통

DOCA Memos가 기존 파일 시스템 스토리지와 다르게 내구성(Durability)을 100% 미만으로 설계한 이유는?

정답: C — KV 캐시는 재연산 가능
KV 캐시는 원본 토큰(텍스트)이 있으면 언제든 재연산(recompute)할 수 있다. 따라서 99.9%에 가까운 내구성으로도 충분히 실용적이며, 이 설계 선택 덕분에 쓰기 증폭(Write Amplification)을 1에 근접하게 낮추고, SSD 수명을 대폭 늘릴 수 있다.
04
객관식 · 단일 선택
● 보통

DOCA Memos의 DPU 측 컴포넌트가 담당하는 주요 역할은?

정답: B — KV 격리·매핑 + 암호화 + 네트워크 전송
컴퓨트 노드의 BlueField-4 DPU에서는 호스트로부터 KV 요청을 받아, 보안 격리(SNAP 기술 기반)·접근 제어·암호화를 적용한다. 이 처리가 호스트 CPU 자원을 전혀 소비하지 않으며, 이후 Spectrum-X 이더넷을 통해 CMX 스토리지 노드로 전송된다. SSD 직접 접근은 CMX 측 BlueField가 담당한다.
05
객관식 · 단일 선택
● 보통

로컬 SSD에 KV 캐시가 분산 저장된 상황(비공유)과 달리, CMX 공유 스토리지를 사용할 때 얻는 핵심 이점은?

정답: D — GPU 활용률 극대화
로컬 SSD 환경에서는 특정 KV 캐시가 특정 노드에만 있어, 해당 요청이 그 노드로만 몰리고 다른 노드는 유휴 상태가 된다. CMX를 통해 모든 KV 캐시를 공유하면 어떤 GPU 노드든 어떤 KV 캐시든 접근 가능하므로, Dynamo 라우터가 부하를 균등 분배해 GPU 활용률을 100%로 유지하고 TTFT(첫 번째 토큰 생성 시간)를 최소화한다.
OX 문제
06
OX 문제
● 쉬움

DOCA Memos는 컴퓨트 노드의 DPU와 스토리지 노드의 BlueField, 양쪽 모두에 소프트웨어가 배포되어 I/O 파이프라인이 분산 처리된다.

정답: ○ (참)
DOCA Memos는 컴퓨트 노드(BlueField-4 DPU)와 CMX 스토리지 노드(BlueField Storage Controller) 양쪽 모두에 실행된다. 각 위치에서 최적의 작업을 수행하도록 I/O 파이프라인이 의도적으로 분리 설계됐다.
07
OX 문제
● 보통

추론 애플리케이션은 DOCA Memos에서 KV 캐시 조회 전에 exists()를 먼저 호출하여 존재 여부를 확인하는 것이 권장 모범 사례다.

정답: ✕ (거짓)
exists() 후 retrieve() 패턴은 두 가지 이유로 지양해야 한다. 첫째, retrieve() 자체가 조회 역할을 하며 없으면 빠르게 실패를 반환한다. 둘째, 내구성이 100% 미만이므로 exists()가 true를 반환해도 실제 retrieve() 시점에는 이미 삭제됐을 수 있다. 불필요한 네트워크 왕복(ping-pong)만 발생시킨다.
08
OX 문제
● 어려움

컨텍스트 "구멍(Holes)" 처리 최적화란, 중간에 특정 KV 블록이 누락되었을 때 컨텍스트 전체를 처음부터 재연산하는 것이 최선이다.

정답: ✕ (거짓)
구멍(holes) 처리의 핵심은 누락된 블록만 재연산하고, 이미 캐시된 나머지 블록은 그대로 활용하는 것이다. 전체 재연산은 CMX의 강점을 완전히 포기하는 것이다. 발표에서는 VLM과 Dynamo가 이를 이미 지원한다고 밝혔다.
단답형
09
단답형
● 보통

DOCA Memos 시뮬레이션에서 18 페타바이트 CMX와 1,000개 GPU 조건 기준, KV 캐시 히트율과 GPU Goodput은 각각 몇 % 이상으로 나타났는가? (숫자로 입력, 예: 히트율 XX%, Goodput XX%)

정답: 히트율 99.8% 이상, GPU Goodput 96% 이상
시뮬레이션 조건: GPT-OSS-20B · 1,000 GPUs · 18PB CMX · 700~5,000 동시 에이전트 · 에이전트당 50 스텝 · 스텝당 2K 토큰. Goodput은 GPU가 생성한 토큰 중 진짜 신규 토큰의 비율이다. 96%는 클러스터 낭비가 4% 미만임을 의미한다.
10
단답형
● 어려움

DOCA Memos에서 KV 캐시를 저장할 때 사용하는 '키(Key)'는 무엇으로 구성되는가? 그리고 왜 애플리케이션이 per-KV 메타데이터를 직접 수집해서는 안 되는가? 두 가지를 간략히 서술하라.

정답 해설
① 키 구성: KV 캐시의 키는 해당 KV가 대응하는 토큰 시퀀스의 prefix 해시(prefix_hash)다. 최대 128비트 크기. 같은 키로 저장하고 나중에 동일 키로 조회하면 된다.

② 메타데이터를 앱에서 수집하면 안 되는 이유: CMX는 최대 18 페타바이트 용량을 제공하므로 수백만 개의 KV 블록이 저장될 수 있다. 이 모든 KV에 대한 메타데이터를 앱(호스트)에서 직접 관리하면 DRAM이 고갈된다. DOCA Memos가 내부적으로 메타데이터를 처리하므로 앱은 단순히 key로 저장·조회만 하면 된다.
