Part 01 — 배경

에이전틱 시대가 만들어낸 메모리 위기

AI 시스템이 단순 챗봇에서 에이전트(Agentic AI)로 진화하면서 근본적인 문제가 등장했다. 에이전트는 여러 단계에 걸쳐 작업을 수행하고, 과거 대화와 학습 내용을 '기억'해야 한다. 마치 직원에게 방대한 매뉴얼을 가르쳐 놓았는데, 다음 날 아침 모든 것을 잊어버린다면 의미가 없는 것처럼.

이 요구는 KV 캐시(KV Cache)라는 형태로 구체화된다. KV 캐시는 AI 모델이 처리한 텍스트(토큰)의 중간 연산 결과물을 저장해 두는 공간이다. 대화가 길어지고 에이전트가 많아질수록 이 데이터는 기하급수적으로 불어난다.

단일 AI 워커 하나의 컨텍스트만 해도 최대 200GB에 달한다. 이것이 수백 개의 에이전트로 늘어나면 기존 인프라로는 감당이 불가능하다.

"이건 단순한 데이터가 아닙니다. 그런데 우리는 기존 인프라로 그것을 다루도록 강제받고 있습니다."

— Ariel Kit, DOCA Products Director

Part 02 — 아키텍처

컨텍스트 메모리 계층 구조

기존 메모리/스토리지 계층은 AI 추론을 위해 설계된 것이 아니다. NVIDIA는 이 문제를 해결하기 위해 기존 G1~G4 계층 사이에 새로운 G3.5 계층(CMX)을 정의했다. 아래 각 계층을 탭하면 상세 설명을 볼 수 있다.

컨텍스트 메모리 계층 — 클릭하여 상세 보기
G1
Local GPU HBM · Active KV
GPU 내부 고대역폭 메모리 — 가장 빠르고 비쌈
HBM(High Bandwidth Memory)은 GPU 내부에 탑재된 최고속 메모리다. 현재 활성 연산에 필요한 KV 캐시가 위치하며, 용량이 극히 제한적이다. 에이전틱 AI 워크로드에서는 곧바로 용량 한계에 도달한다.
G2
Local System Memory · Staging KV
CPU DRAM — 중간 속도, 공유 불가
GPU HBM이 넘치면 CPU의 시스템 메모리(DRAM)로 스필오버된다. 최근 키노트에서 강조되는 더 강력한 CPU 메모리가 여기에 해당한다. 하지만 여러 에이전트가 긴 대화를 동시에 처리하면 이 계층도 금방 부족해진다.
G3
Local Storage · Warm KV Reuse
로컬 SSD — 느리고 공유 불가
로컬 NVMe SSD에 KV 캐시를 저장할 수 있지만, 두 가지 결정적인 단점이 있다. 첫째, HBM보다 훨씬 느리다. 둘째, 다른 GPU/노드와 공유할 수 없어 전체 클러스터 활용률이 낮아진다.
G3.5
🆕 NVIDIA CMX · POD Shared KV Cache Storage
AI 전용 공유 스토리지 — 신규 계층
CES 2026에서 처음 발표된 신규 스토리지 계층이다. POD(컴퓨트 노드 묶음) 내 모든 GPU가 공유 접근할 수 있으며, 컴퓨트 노드 인근에 위치해 지연을 최소화한다. 용량은 최대 18 페타바이트. BlueField-4 DPU, Spectrum-X Ethernet, DOCA, Dynamo로 가속된다. 기존 스토리지 대비 토큰 처리량 5배, 전력 효율 5배를 달성한다.
G4
Data Center Network Storage · Cold KV
원거리 네트워크 스토리지 — 고비용, 고지연
데이터센터 전체에서 접근 가능한 네트워크 스토리지다. 컴퓨트와 물리적으로 멀어 지연이 크다. 발표에서는 이를 "소금이 없어서 슈퍼마켓에 차를 몰고 가는 것"에 비유했다 — 차라리 처음부터 다시 요리를 시작하는 게 나을 수도 있다. GPU가 재연산(recompute)을 택하는 이유이며, 이는 전력 낭비로 이어진다.
Part 03 — 솔루션

NVIDIA CMX: AI 전용 스토리지 플랫폼

CMX(Context Memory eXtension)는 GPU 메모리의 확장으로 설계되었다. 단순한 스토리지 제품이 아니라, NVIDIA가 설계하고 스토리지 파트너사들이 제조하는 공동 최적화(co-optimized) 솔루션이다.

5×
토큰 처리량 향상
(기존 대비)
5×
전력 효율 향상
(Tokens per Watt)
주요 파트너사: AIC, Cloudian, DDN, Dell Technologies, HPE, Hitachi Vantara, IBM, NetApp, Nutanix, Pure Storage, SuperMicro, VAST, WEKA
시뮬레이션 — 공유 vs 비공유 KV 캐시 스토리지 (100 GPUs)

파라미터: 100 GPUs · 에이전트당 64 스텝 · 스텝당 4K 토큰 · 128K 캐시 토큰

공유 스토리지(파란선)는 동일한 GPU 활용률 100%를 더 넓은 TPS/User 범위에서 유지한다. 같은 375명의 동시 사용자를 기준으로, 공유 스토리지는 훨씬 높은 토큰/초를 제공하거나, 동일한 서비스 품질로 최대 750명(약 2배)의 사용자를 처리할 수 있다.

Part 04 — 소프트웨어

DOCA Memos: 에이전틱 시대의 새 SDK

하드웨어(CMX)만으로는 부족하다. 이를 활용하는 소프트웨어가 필요하다. NVIDIA는 GTC 2026에서 DOCA Memos를 공식 발표했다. DOCA 4.0의 새로운 SDK로, KV 캐시 가속과 공유를 POD 스케일에서 처리한다.

왜 새로운 스토리지 프로토콜인가?

가장 효율적인 스토리지 프로토콜은 블록(Block)이지만, 블록은 공유가 어렵다. 반면 파일 시스템이나 오브젝트 스토리지는 공유는 가능하지만 복잡성이 높고 성능이 떨어진다. DOCA Memos는 그 중간 — 블록 수준의 성능에 공유 가능성을 더한 새로운 스토리지 클래스다.

특성 DOCA Memos 기존 파일/오브젝트 스토리지
I/O 크기 대형(>1MB), 고정, 사전 설정 1B~수 GB, 유연
변경 가능성 불변(Immutable) 어느 위치든 수정 가능
내구성 <100% (의도적) 100% 보장
메타데이터 최소화 파일명, 폴더, 키, lookup 등 방대
쓰기 증폭 1에 근접 높음
SSD 수명 대형 I/O로 SSD 친화적 범용 목적

내구성이 100% 미만인 것은 설계상의 선택이다. KV 캐시는 잃어버려도 원본 토큰에서 재연산할 수 있다(recomputable). 따라서 99.9% 신뢰성이면 충분하며, 그 덕분에 쓰기 증폭을 극적으로 낮추고 SSD 수명을 연장할 수 있다.

시뮬레이션 — DOCA Memos 히트율 (1,000 GPUs · 18PB CMX)

파라미터: GPT-OSS-20B · 700~5,000 동시 에이전트 · 에이전트당 50 스텝 · 스텝당 2K 토큰

KV 캐시 히트율
시뮬레이션 결과
>96%
GPU Goodput
(순수 신규 토큰 생성 비율)
Goodput(굿풋)이란? GPU 활용률(토큰/초)에서 재연산된 토큰을 뺀 값이다. 캐시에 저장할 수 있었지만 GPU가 다시 계산한 토큰은 낭비다. 96% Goodput은 GPU 클러스터를 거의 100% 효율로 활용한다는 의미다.
Part 05 — 구성요소

DOCA Memos 3계층 아키텍처

DOCA Memos는 컴퓨트 노드(AI 추론 서버)와 스토리지 노드(CMX) 양쪽 모두에 배포된다. 각 위치에서 최적의 처리를 담당하도록 I/O 파이프라인이 분리 설계되었다.

DOCA Memos 데이터 흐름 — 클릭하여 상세 보기
HOST
Host (컴퓨트 노드) — DOCA Key Value Driver
추론 앱에 C API 제공 · KV 요청 생성
추론 애플리케이션(vLLM, SGLang 등)에 store(key, val) / retrieve(key) C API를 노출한다. key는 토큰의 prefix 해시값이며, val은 KV 캐시 블록이다. 이 드라이버는 IO 요청을 DPU로 전달한다.
DPU
BlueField-4 DPU — DOCA Memos Initiator
KV 격리·매핑 · 암호화 · 네트워크 전송
컴퓨트 노드의 DPU(Data Processing Unit)에서 실행된다. 호스트의 KV 요청을 받아 보안 격리(isolation), 접근 제어, 암호화를 적용한다. 그 후 Spectrum-X 이더넷을 통해 CMX 스토리지 노드로 네트워크 요청을 보낸다. 이 모든 처리가 호스트 CPU 컴퓨트를 전혀 소비하지 않는다.
CMX
CMX Storage Node — DOCA Memos Target
데이터 배치 · SSD 직접 읽기/쓰기
CMX 스토리지 노드의 BlueField 스토리지 컨트롤러에서 실행된다. 네트워크로부터 요청을 받아 최종적으로 SSD에 블록을 쓰거나 읽는다. API: write_blocks_cb(data_ptr, disk, lba, len) / read_blocks_cb(...). 데이터 인접 처리로 최적 효율을 달성한다. 18 페타바이트 규모에서 메타데이터 관리를 앱이 아닌 CMX 자체에서 처리한다.
Part 06 — 인퍼런스 스택 통합

NIXL: 이기종 데이터 전송을 하나로

NIXL(NVIDIA Inference Xfer Library)은 GTC 2025에서 출시된 AI 추론용 범용 데이터 전송 라이브러리다. GPU↔GPU, 메모리↔스토리지 등 다양한 이기종 데이터 이동을 단일 API로 추상화한다.

NIXL V1.0이 이번 주 정식 출시됐다. pip install nixl로 즉시 설치 가능. TRT-LLM, vLLM, SGLang, Dynamo KVBM, LMCache, AnyScale 등 주요 추론 프레임워크에 기본 통합됐다. 이번 GTC에서 CMX(DOCA Memos) 지원이 새롭게 추가됐다.

지원 패브릭: RDMA, NVLink, AWS Trainium/Inferentia, Broadcom, AWS EFA, Slingshot. 지원 스토리지: 블록, S3, S3/RDMA, Azure Blob, DeepSeek 3FS, GCP (예정), CMX via DOCA Memos (신규).

Part 07 — 개발자 가이드

DOCA Memos 통합 모범 사례

기존 추론 프레임워크가 DOCA Memos와 통합될 때 주의해야 할 핵심 사항들이다.

  • 1
    키는 최대 128비트 사용. prefix 해시를 key로, KV 블록을 value로 저장한다. store(key, val) / retrieve(key)
  • 2
    앱에서 per-KV 메타데이터를 직접 수집하지 말 것. CMX는 사실상 무한(18PB)에 가까우므로, 모든 KV 메타데이터를 앱에서 추적하면 DRAM을 고갈시킨다. DOCA Memos가 대신 처리한다.
  • 3
    Huge Pages를 사용할 것. KV 블록은 수 MB 단위로 크다. 4KB 소형 페이지를 쓰면 수천 번의 페이지 매핑이 발생해 큰 오버헤드가 생긴다.
  • 4
    exists() 후 retrieve() 패턴을 피할 것. retrieve()가 이미 조회 역할을 한다. 게다가 exists()가 true를 반환해도 retrieve() 시점에 이미 삭제됐을 수 있다 (100% 내구성 미보장).
  • 5
    실패한 retrieve()를 정상 처리할 것. KV가 없을 경우 다음 계층(G4) 검색 또는 재연산으로 폴백해야 한다.
  • 6
    컨텍스트 "구멍(holes)" 계산을 지원할 것. 컨텍스트 중간의 특정 KV 블록만 누락될 수 있다. 누락된 블록만 재연산하고 나머지는 재사용하는 로직이 필요하다. VLM과 Dynamo에서 이미 지원 중.
  • 7
    라우팅 함수를 수정할 것. CMX를 쓰면 KV 캐시가 어느 노드에 있든 상관없이 접근 가능하다. Dynamo 라우터가 이를 반영해 최적 노드로 요청을 분배해야 한다.
